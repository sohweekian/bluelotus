STATUS: RESOLVED

================================================================================
1. EXECUTIVE SUMMARY
================================================================================

During continuation of the S8 Global Leverage Unwind Thesis widget work order,
S8 was successfully implemented, tested, pushed, and started as an independent
V3 thesis widget daemon.

However, during dashboard card deployment, S8 was inserted into the then-current
GitHub Pages index.html that had been regenerated by the V3 publisher and did
not contain the previously injected S6 and S7 cards. As a result, the public
dashboard showed S8 but S6 and S7 disappeared.

The issue has been corrected. S6, S7, and S8 are now all present on the public
homepage and their JSON endpoints are live.

================================================================================
2. ROOT CAUSE
================================================================================

Primary root cause:

  Dashboard card injection was performed against a publisher-generated
  index.html snapshot that did not include the previously injected S6/S7 card
  blocks.

Architectural cause:

  The dashboard currently has two competing homepage writers:

  1. bluelotus_publisher.py
     - Regenerates and pushes index.html.
     - Does not yet permanently own S6/S7/S8 widget-card composition.

  2. scripts/update_s6_card.py / update_s7_card.py / update_s8_card.py
     - Inject card blocks into the current GitHub index.html.
     - Works, but is vulnerable if publisher later overwrites index.html.

Operational mistake:

  S8 was deployed into the currently visible/publisher-generated dashboard shell
  without first verifying that S6 and S7 markers were present in that same HTML
  source. That allowed S8 to be preserved while S6/S7 remained absent.

Secondary issue:

  GitHub Pages legacy build failed until .nojekyll was added. The repository
  copy of index.html contained S8, but Pages initially served an older built
  page. Adding .nojekyll fixed the Pages build status.

7. PERMANENT ENGINEERING LESSON
================================================================================

Widget cards must not be deployed into an index.html snapshot unless the full
expected card set is verified first.

Before any future S6/S7/S8/S9 card update, the deployer must verify the current
GitHub Contents API copy of index.html contains all expected markers:

  ai-infra-live
  credit-refi-live
  global-leverage-live

If any marker is missing, the deployer must either:

  1. Reapply all widget cards in order, or
  2. Stop and repair the dashboard shell source of truth.

The correct order for current widget reinjection is:

  S6 first
  S7 second
  S8 third

================================================================================
8. RECOMMENDED ARCHITECTURAL FIX
================================================================================

The current injection approach works but is not robust against publisher
regeneration. The permanent fix should be one of the following:

Option A - Publisher owns widget slots

  Modify bluelotus_publisher.py so the generated homepage includes stable
  widget slots for S6/S7/S8 and future S9/S10/S11 widgets.

Option B - Widget card registry assembler

  Create a dashboard widget registry file, for example:

    C:\bluelotus3\config\dashboard_widgets.yaml

  Then make a single script assemble all enabled widget cards into index.html.

Option C - Dedicated placeholder container

  Add a permanent marker block to publisher output:

    <!-- BLUELOTUS_THESIS_WIDGETS_START -->
    <!-- BLUELOTUS_THESIS_WIDGETS_END -->

  Injection scripts may only replace content inside that bounded block.

Recommended choice:

  Option C immediately, then Option B when widget count grows beyond S8.
