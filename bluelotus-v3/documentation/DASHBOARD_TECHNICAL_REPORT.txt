0. ENGINEERING DOCTRINE — READ THIS FIRST
================================================================================

  This section is non-negotiable. Every contributor and every AI assistant
  working on BlueLotus V3 must read and honour these rules before touching
  any file in this repository.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  RULE 1 — NO HARDCODING. EVER.                                          │
  │                                                                         │
  │  Configuration values — URLs, thresholds, time windows, intervals,      │
  │  limits, labels, ticker lists, scoring weights, keyword lists — must    │
  │  NEVER appear as literals inside Python (.py) files.                    │
  │                                                                         │
  │  Every configurable value lives in a JSON config file.                  │
  │  Python reads it. Python never owns it.                                 │
  │                                                                         │
  │  Correct:   freshness_min comes from news_probe_sources.json            │
  │  WRONG:     freshness_min = 720  inside news_probe_daemon.py            │
  │                                                                         │
  │  To change a setting → edit the JSON → restart the daemon.             │
  │  Never modify Python code for a configuration change.                   │
  └─────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  RULE 2 — MODULAR DESIGN. ONE DAEMON, ONE RESPONSIBILITY.               │
  │                                                                         │
  │  Each daemon is a fully independent module. It has its own:             │
  │    - Script (.py)                                                        │
  │    - Config file(s) (.json)                                             │
  │    - Log file (.log)                                                    │
  │    - Output JSON pushed to GitHub Pages                                 │
  │                                                                         │
  │  Daemons do NOT call each other. They do NOT share state.              │
  │  They communicate only through JSON files on disk or GitHub Pages.      │
  │                                                                         │
  │  Killing one daemon does not affect any other daemon.                   │
  │  Restarting one daemon does not require touching any other daemon.      │
  └─────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  RULE 3 — CONFIG FILES ARE THE SINGLE SOURCE OF TRUTH.                  │
  │                                                                         │
  │  If a value appears in both a JSON config AND a Python file, the        │
  │  Python file is WRONG. Delete the Python copy. Read from JSON.          │
  │                                                                         │
  │  Config files must validate their own required fields at daemon          │
  │  startup. If a required field is missing, the daemon refuses to start   │
  │  with a FATAL error — it does NOT fall back to a hidden default.        │
  └─────────────────────────────────────────────────────────────────────────┘

  │  RULE 4 — DAEMONS NEVER EXIT ON RUNTIME ERRORS.                         │
  │                                                                         │
  │  Network failures, API timeouts, GitHub push failures, bad RSS feeds   │
  │  — these are expected conditions. The main loop catches all exceptions, │
  │  logs them, and continues to the next cycle.                            │
  │                                                                         │
  │  A daemon exits ONLY on startup fatal errors (bad config file,          │
  │  missing required fields) or explicit Ctrl-C by the operator.           │
  └─────────────────────────────────────────────────────────────────────────┘

================================================================================
1. OVERVIEW
================================================================================

  The BlueLotus V3 dashboard is a GitHub Pages static site powered by five
  independent long-running Python daemons on this Windows machine. Each daemon
  runs on a fixed interval, computes its output, and pushes a JSON file to the
  GitHub repository. The browser JavaScript fetches those JSON files to render
  live dashboard sections.

  No web server. No MySQL. No pipeline dependency. Each daemon is a fully
  self-contained module — its own config, its own log, its own output.

  All five daemons are launched by a single batch file:

      C:\bluelotus3\start_daemons.bat

  Python interpreter (bluelotus3 has no venv — falls back to V2 venv):

      C:\bluelotus2\.venv\Scripts\python.exe

  Credentials file:

      C:\bluelotus3\.env
      (contains GITHUB_TOKEN, GITHUB_USERNAME, GITHUB_PAGES_REPO,
       GITHUB_BRANCH, TELEGRAM_BOT_TOKEN, TELEGRAM_CHAT_ID)

  IMPORTANT: Daemons read GITHUB_TOKEN at module load (startup). If the
  .env file is empty or missing when the daemon starts, the token will be
  blank for the entire lifetime of that process. Fix: populate .env first,
  then start daemons.

4. MODULAR ARCHITECTURE — HOW IT ALL FITS TOGETHER
================================================================================

  Each daemon is a module. Modules are independent. This is not accidental —
  it is a deliberate design principle of the BlueLotus V3 digital organisation.

  SEPARATION OF CONCERNS:
  +-----------------------+----------------------------------------------------+
  | Layer                 | What lives here                                    |
  +-----------------------+----------------------------------------------------+
  | .env                  | Secrets only (tokens, credentials). Never config.  |
  | *_sources.json        | Source registry: IDs, URLs, labels, logos,         |
  |                       | per-source thresholds and freshness windows.       |
  | *_config.json         | Runtime settings: intervals, limits, toggles.      |
  | *_daemon.py           | Logic only. Reads config. Computes. Pushes.        |
  |                       | Contains ZERO hardcoded configuration values.      |
  | data/*.json           | Output. Written by daemons. Read by JS dashboard.  |
  +-----------------------+----------------------------------------------------+

  WHY THIS MATTERS:
  - A non-engineer (or a different AI assistant next session) can change a
    freshness window, add a news source, or adjust a scoring threshold by
    editing a JSON file in a text editor — no Python knowledge required.
  - Config changes are version-controlled, diffable, and reviewable.
  - Python files change only when logic changes — not when parameters change.
    This means fewer Python restarts, fewer logic bugs, and a clean audit trail.
  - Modules fail loudly on misconfiguration (FATAL at startup) rather than
    silently using stale defaults that produce wrong dashboard data.

  INTER-DAEMON COMMUNICATION:
  Daemons share data only through JSON files on disk:
    - Warsh Thesis Probe reads headlines_live.json  (written by News Probe)
    - BOJ Engine reads headlines_live.json           (written by News Probe)
  There are no direct function calls, no shared memory, no message queues.
  This keeps every daemon independently restartable.

  ADDING A NEW NEWS SOURCE (example of how modular design works in practice):
  1. Add an entry to news_probe_sources.json with label, url, method,
     freshness_min, and logo.
  2. Add the source ID to source_order in news_probe_sources.json.
  3. Restart news_probe_daemon.py.
  That is all. Zero Python changes. The new source appears on the dashboard
  automatically because the JS reads source_order dynamically from the JSON.

9. ENGINEERING CHECKLIST — FOR ANY FUTURE CODE CHANGES
================================================================================

  Before modifying any daemon or config file, ask these questions:

  [ ] Is this a config change or a logic change?
      - Config change → edit JSON only, restart daemon, do not touch Python.
      - Logic change  → edit Python only, no values in the code.

  [ ] Does the Python file I am changing contain any hardcoded values?
      (URLs, numbers, thresholds, strings that could ever need changing)
      If yes → move them to the appropriate JSON config file first.

  [ ] Does my change affect other daemons?
      - If it changes a JSON output schema → update all readers.
      - If it adds a new JSON field → ensure backward compatibility.

  [ ] Does the daemon validate its config at startup?
      - Any new required config field must be validated at startup.
      - Startup FATAL errors are better than silent wrong values at runtime.

  [ ] Did I update this technical report?
      - Source table changes → Section 3.
      - New config fields → Section 3 runtime config block.
      - New files → Section 5.
      - Status changes → Section 6.

10.4  SCORING LOGIC
--------------------------------------------------------------------------------

  STEP 1 — Fetch prices (yfinance)
    Batch download: all 24 thesis tickers + SPY + QQQ in one call.
    Handles new yfinance MultiIndex column structure (ticker, field).
    Graceful per-ticker failure: missing data → signal = UNKNOWN.
    data_quality: FULL (all) / PARTIAL (some) / UNAVAILABLE (none).

  STEP 2 — Score each basket
    For each ticker in basket:
      signal = PASS  if day_change_pct >= pass_pct
      signal = FAIL  if day_change_pct <= fail_pct  (inclusive boundary)
      signal = WATCH otherwise
    basket_score = (PASS_count / total_enabled_tickers) * basket_weight
    basket_signal = PASS if >50% PASS, FAIL if >50% FAIL, else WATCH

  STEP 3 — Score headlines
    Read data/headlines_live.json (if available; degrade gracefully if missing)
    Scan all headline text for 25 config-defined keywords
    Each unique keyword match contributes: keyword_score = headline_max/len(keywords)
    headline_score = sum(unique matches) capped at headline_max

  STEP 4 — Risk penalty
    If SPY day_change_pct < spy_threshold AND QQQ day_change_pct < qqq_threshold:
      penalty = min(risk_penalty_max, |avg(SPY,QQQ)| * multiplier)
    Penalty is subtracted from total score.

  STEP 5 — Total score and status
    total = basket_total + headline_score - risk_penalty
    status = map(total → CONFIRMING/WATCH/MIXED/WEAKENING/CONTRADICTED)
    confidence = map(data_fraction → HIGH/MEDIUM/LOW/UNKNOWN)
    cio_action = cio_action_map[status + "_" + confidence]
    add_allowed = (status == "CONFIRMING" and no concentration override)

10.12  KNOWN LIMITATIONS
--------------------------------------------------------------------------------

  1. MARKETS CLOSED DETECTION
     When US markets are closed (nights, weekends), yfinance returns the
     previous session's closing prices. Day_change_pct reflects Monday→Friday
     close, not an intraday move. The widget does not flag this.
     Recommended next step: add market_open flag to JSON output.

  2. HEADLINE SCORE ONLY WHEN DAEMON RUNNING
     If news_probe_daemon.py is not running, data/headlines_live.json is stale
     or absent. The widget handles this gracefully (headline_score = 0) but the
     headline component is dark. Not a bug — by design (modular independence).

  3. SMR TICKER
     SMR (NuScale Power) has very low liquidity. yfinance may return stale or
     zero volume data on some days. The widget handles it gracefully (UNKNOWN)
     but it does not weight-adjust for liquidity.

  4. BUCKET-LEVEL RISK PENALTY
     The risk penalty currently applies only to the combined SPY+QQQ drop.
     It does not detect sector-specific AI sell-offs that leave SPY flat.
     Recommended next step: add AI-specific liquidation check (NVDA + AMD > 5%
     simultaneous drop triggers ai_sector_penalty).
