1. EXECUTIVE SUMMARY
================================================================================

The BlueLotus V3 dashboard is a live, registry-driven, multi-thesis intelligence
dashboard hosted on GitHub Pages. It surfaces four independent thesis widgets
(S6 through S9) and a V2-legacy CIO intelligence layer on a single public
homepage.

Key architectural facts:

  - Four standalone thesis widget daemons run independently of each other and of
    the V3 publisher pipeline. Each daemon fetches market prices via yfinance,
    scores a set of baskets against configurable YAML thresholds, and pushes a
    JSON result to GitHub every 600 seconds.

  - The dashboard (index.html) is rendered by bluelotus_publisher.py and hosted
    on GitHub Pages. Widget cards are registry-driven HTML+JS blocks injected
    into a protected zone; they self-refresh in the browser via fetch() every
    600 seconds without requiring a new HTML push.

  - A central dashboard widget registry (config/dashboard_widgets.yaml) is the
    single source of truth for which widgets exist, their render order, their
    marker IDs, and their JSON endpoints. The publisher reads the registry at
    build time and refuses to publish an index.html that is missing any enabled
    widget marker (fail-closed guard).

  - All four widgets enforce: CIO_ONLY_MANUAL execution authority,
    order_routing_enabled = false, llm_order_generation = false, no broker
    imports, no LLM imports (Qwen, Ollama, OpenAI, Anthropic). These invariants
    are enforced by the test suite on every build.

  - Test count as of 2026-06-17: 339 passed, 2 warnings (unrelated to BlueLotus).

3. THESIS WIDGET DAEMON ARCHITECTURE
================================================================================

Each of the four V3 thesis widgets (S6–S9) follows an identical architectural
pattern. The pattern is a contract, not a convention — tests enforce it.

3.1 STRUCTURAL INVARIANTS (enforced by test suite)
────────────────────────────────────────────────────

  NO_HARDCODING_DOCTRINE
    All operational values live exclusively in the YAML config file. Python
    reads the config at startup and owns zero operational constants. This
    includes: ticker symbols, scoring weights, thresholds, market hours,
    timezone, output paths, GitHub paths, headline keywords, status strings,
    CIO action labels, and safety constants. Consequence: any operational
    change is a config edit followed by a daemon restart — never a code change.

  SAFETY CONSTANTS (identical across all four widgets)
    execution_authority:   "CIO_ONLY_MANUAL"
    order_routing_enabled: false
    llm_order_generation:  false
    orders_generated:      0        (always 0 in every output)

  FORBIDDEN IMPORTS
    None of the following may appear anywhere in a V3 widget Python file:
    openai, anthropic, ollama, qwen, langchain, alpaca, ibapi, ib_insync,
    moomoo, futu. Tests parse Python ASTs to enforce this.

  FORBIDDEN PATHS
    No reference to "C:\bluelotus2" in any V3 widget Python file. V3 and V2
    are fully separated systems. Tests enforce this.

  SCHEMA
    Every widget output JSON carries: schema_version = "thesis_widget_v1.1",
    thesis_id, status, score, confidence, cio_action, add_allowed, risk_level,
    data_quality, market_status, market_open, last_updated_sgt, last_updated_utc,
    primary_signals, ticker_evidence, headline_score, external_evidence,
    blind_spots, safety block. Tests validate all required fields on every run.

3.2 EXECUTION LOOP
───────────────────

  Each daemon runs a continuous loop:

    while True:
        run_once(cfg)     # fetch prices, score, build output, save, push to GitHub
        sleep(600)        # refresh_interval_seconds from YAML

  Each run_once() call:
    1. Fetch market prices for all configured tickers via yfinance (5-day window)
    2. Compute day_change_pct for each ticker
    3. Score each basket (weighted pass/watch/fail counts → basket score)
    4. Apply basket-level signal (pass/fail/watch/stress/calm depending on widget)
    5. Score headline feed (data/headlines_live.json) via keyword matching
    6. Apply escalation or risk penalties where configured
    7. Apply false-positive rules where configured (S9 only)
    8. Read external evidence from peer widget JSON files (optional, graceful)
    9. Compute total score (clamped 0–100)
    10. Map score → status → confidence → cio_action
    11. Compute add_allowed from status + confidence + basket gates + external evidence
    12. Build full output dict
    13. Save JSON locally
    14. Push JSON to GitHub via REST API (base64-encoded content, creates or updates file)

3.3 DATA QUALITY TRACKING
──────────────────────────

  Each widget tracks yfinance availability per ticker:
    FULL:        all symbols returned valid price data
    PARTIAL:     some symbols returned valid data, some failed
    UNAVAILABLE: no symbol returned valid data

  data_quality is surfaced in the JSON output and is visible in the dashboard
  card. PARTIAL and UNAVAILABLE degrade the score and trigger heightened CIO
  review recommendations.

3.4 MARKET HOURS AWARENESS
────────────────────────────

  Each widget determines its market_status at run time from the YAML config:
    OPEN:           market is live (09:30–16:00 ET, Mon–Fri)
    CLOSED_PRE:     before market open on a trading day
    CLOSED_POST:    after market close on a trading day
    CLOSED_WEEKEND: Saturday or Sunday

  The data_freshness_label reflects this in the dashboard card, so the CIO
  always knows whether the data is intraday or last-session close.

3.5 EXTERNAL EVIDENCE (WIDGET CROSS-READS)
────────────────────────────────────────────

  S8 reads S5 (BOJ/Yen Watcher) and S7 (Credit/Liquidity) JSON files as
  optional external evidence. Absent files are handled gracefully — no crash,
  no hard dependency.

  S9 reads S7 (Credit/Liquidity) and S8 (Leverage Unwind) JSON files as
  optional external evidence. S7 ACTIVE_STRESS or SEVERE_STRESS blocks
  add_allowed. S8 WATCH, ACTIVE_UNWIND, or SEVERE_UNWIND blocks add_allowed.

  External evidence is strictly read-only and optional. No widget writes to
  another widget's data file or calls another widget's code.

3.6 HEADLINE FEED
──────────────────

  All four widgets read data/headlines_live.json produced by the V3 news probe
  daemon (news_probe_daemon.py). If the file does not exist, headline_score is
  zero and the widget continues without error. Headline scoring is purely
  deterministic: case-insensitive substring keyword matching — no LLM, no NLP.

  S9 additionally scans for escalation keywords (e.g., "Hormuz closed",
  "tanker attack", "missile strike") and applies a negative score penalty
  (max -25) to prevent false-positive peace-dividend signals during
  re-escalation events.

6.2 DASHBOARD WIDGET MANAGER
──────────────────────────────

  File: C:\bluelotus3\scripts\dashboard_widget_manager.py

  Key functions:

    load_registry(path)
      Loads and validates the YAML registry. Raises on schema violations.

    validate_registry(registry)
      Enforces all structural invariants: unique marker IDs, unique section IDs,
      required fields on every widget entry.

    enabled_widgets(registry)
      Returns the enabled widget list sorted by display_order.

    render_widget_card(widget, registry)
      Renders the full HTML+JS block for one widget. Reads colour palette,
      JSON endpoint, refresh interval, and all display parameters from the
      registry entry. Returns a self-contained string of HTML.

    render_widget_zone(registry)
      Renders the full protected zone: START marker + all enabled cards in
      display_order + END marker. Returns the complete HTML block.

    verify_html(html_text, registry)
      Validates a candidate index.html string:
        - START marker present exactly once
        - END marker present exactly once
        - Zone appears before "System Health" text
        - All enabled widget markers present in HTML
        - No disabled widget markers present
        - No unexpected duplicate marker references
      Returns a list of error strings (empty = valid).

    replace_or_insert_zone(html_text, registry, restore=False)
      Normal mode (restore=False): replaces the existing zone between markers.
        Raises ValueError if markers are absent.
      Restore mode (restore=True): inserts a fresh zone before "System Health"
        if no zone markers are present. Used by recovery scripts.

    assert_publishable(html_text, registry)
      Calls verify_html(). If errors are found AND the registry's
      FAIL_PUBLISH_IF_ENABLED_WIDGET_MISSING flag is set (default: true),
      raises RuntimeError. This prevents a partial dashboard from being pushed
      to GitHub.

    ensure_nojekyll(root)
      Creates the .nojekyll file in the repo root if absent.

6.3 PUBLISHER INTEGRATION
───────────────────────────

  bluelotus_publisher.py is wired to the widget manager at three points:

    1. After building the base index.html HTML:
       zone_html = render_widget_zone(registry)
       html = replace_or_insert_zone(html, registry)

    2. Before pushing to GitHub:
       assert_publishable(html, registry)
       # Raises RuntimeError if any enabled widget marker is missing

    3. Before pushing:
       ensure_nojekyll(repo_root)
       # Guarantees .nojekyll is always present

  This means: every time bluelotus_publisher.py runs and pushes index.html,
  the S6–S9 widget zone is rebuilt from the registry and verified before push.
  It is structurally impossible for a publisher run to silently drop a widget.

12. KNOWN LIMITATIONS AND BLIND SPOTS (SYSTEM-WIDE)
================================================================================

  1. All equity and ETF price data reflects the last US market session close
     when the NYSE is closed. No pre-market or overnight data.

  2. yfinance is the sole market data source. It is not a licensed real-time
     feed and may have delays, gaps, or occasional symbol outages.

  3. Headline matching is deterministic keyword substring search — no semantic
     analysis, no disambiguation, no sentiment scoring. A headline mentioning
     "no deal reached on ceasefire" would score positively on "deal reached"
     and "ceasefire". The keyword lists must be maintained with care.

  4. The news headline feed (data/headlines_live.json) depends on
     news_probe_daemon.py running continuously. If the daemon is stopped,
     headline_score is zero for all widgets until it restarts.

  5. No intraday Japan market data. EWJ and DXJ reflect the prior US session
     close even when the Tokyo Stock Exchange is open.

  6. No direct futures data (Brent, WTI, CBOT). USO and BNO are ETF proxies.

  7. No repo rate, SOFR, or TONAR monitoring. Dollar funding stress is inferred
     from UUP (ETF) only.

  8. No prime broker data, hedge fund positioning data, or COT data in V3
     widgets (these exist in V2 pipeline but are separate systems).

  9. CryptoLiquidation basket (S8) uses BTC-USD and ETH-USD only. No
     on-chain data, no funding rate, no open interest monitoring.

  10. add_allowed = true is a necessary but not sufficient condition for the
      CIO to add risk. It is a signal that the system's quantitative gates are
      open, not a directive. CIO manual judgment is always required.
