------------------------------------------------------------------------------
  DEFECT-07 — moomoo_intelligence.py: Pre/Post Market Price Capture Failure
  Severity      : CRITICAL
  Classification: Data integrity — live price pipeline
  Owner         : MID Engineering
  Target        : Friday 5 June EOD — before any Phase 2 features
  Effort        : 1 day
------------------------------------------------------------------------------

Problem:
  moomoo_intelligence.py currently captures only the regular US trading
  session closing price for all tickers. It does not capture pre-market
  or post-market prices.

  The US regular trading session runs 09:30 - 16:00 ET, which is
  21:30 - 04:00 SGT. This means the live_prices data in dataset_raw.json
  is sourced from the correct real-time feed for only approximately
  6.5 hours per 24-hour cycle.

  For the remaining ~17.5 hours per day — which covers:
    Pre-market   : 04:00 - 09:30 ET  (16:00 - 21:30 SGT)
    Post-market  : 16:00 - 20:00 ET  (04:00 - 08:00 SGT)

  moomoo_intelligence.py serves the PREVIOUS regular session close price.
  This is stale, wrong, and misleading for all 24/7 portfolio valuation
  and signal purposes.

  Confirmed example from live screenshot (2026-06-04 16:11 SGT):
    WFC regular session close : $78.680  (-$0.760, -0.96%)
    WFC pre-market price      : $79.047  (+$0.367, +0.47%)
    moomoo_intelligence.py    : $78.680  ← WRONG (stale close)
    Error on 40 shares        : $14.68 understatement of market value

  This error compounds across every position in the portfolio and
  affects:
    - live_prices[ticker].price in dataset_raw.json
    - Portfolio market value calculation
    - Unrealised P/L per position
    - Total fund NAV
    - 8-Lens L2 technical scoring (uses current price vs MAs)
    - CIO portfolio compression section

  The CIO has observed this problem for multiple days. The report
  is presenting incorrect portfolio valuations during the majority
  of the Singapore trading day — which is when the CIO is most
  active.

Required fix:

  moomoo_intelligence.py must detect the current market session state
  and request the appropriate price field from the Moomoo API:

  Session detection logic:
    current_et = convert SGT to ET
    if 09:30 <= current_et < 16:00:
        session = "REGULAR"
        price_field = get_market_snapshot() regular price
    elif 04:00 <= current_et < 09:30:
        session = "PRE_MARKET"
        price_field = get_market_snapshot() pre-market price
    elif 16:00 <= current_et < 20:00:
        session = "POST_MARKET"
        price_field = get_market_snapshot() after-hours price
    else:
        session = "CLOSED"
        price_field = last known close (with CLOSED flag)

  The live_prices export must include:
    price         : the most current available price
                    (pre/regular/post depending on session)
    session       : "REGULAR" / "PRE_MARKET" / "POST_MARKET" / "CLOSED"
    regular_close : last regular session close (always stored)
    pre_mkt_price : pre-market price if available (null if not)
    post_mkt_price: post-market price if available (null if not)
    price_source  : which field is being used as the primary price

  The research report must display session context alongside price:
    WFC  $79.047  [PRE-MARKET]  (+0.47% pre-mkt)

  The portfolio position table must use the session-correct price
  for unrealised P/L and market value calculations.

  Data source verification required:
    Confirm which Moomoo API field returns pre-market price.
    Confirm which field returns post-market price.
    Verify against official Moomoo documentation before implementing.
    Do NOT assume field names — confirm them.
    If Moomoo API does not provide pre/post market prices,
    document this limitation explicitly and flag to CIO.

Required addition to dataset_raw.json:
  meta.market_session : current session state at time of export
  live_prices[ticker].session : session state for this price capture
  live_prices[ticker].price   : session-correct current price
  live_prices[ticker].regular_close : last regular session close

Required addition to research report:
  Dataset integrity section must show:
    Market Session    : PRE_MARKET / REGULAR / POST_MARKET / CLOSED
    Price Source      : pre-market / regular session / post-market

  If session = PRE_MARKET or POST_MARKET, add warning:
    "Prices reflect [pre/post]-market data. Lower liquidity.
     Regular session close also available."

  If session = CLOSED, add warning:
    "Markets closed. Prices reflect last regular session close.
     Pre/post market data not available for this cycle."

Acceptance criteria:
  live_prices[ticker].price correctly reflects pre-market pricing
  during SGT 16:00 - 21:30 window.
  live_prices[ticker].price correctly reflects post-market pricing
  during SGT 04:00 - 08:00 window.
  Portfolio market value and unrealised P/L calculated from
  session-correct price in all report cycles.
  meta.market_session field present and correct in every export.
  Research report shows session label alongside all prices.

  P2-01 — Short Interest and Squeeze Risk Scoring
  Priority      : CRITICAL — PRIMARY CIO REQUIREMENT
  Owner         : MID Engineering
  Target        : Saturday 6 June EOD
  Effort        : 2 days
------------------------------------------------------------------------------

Problem:
  QBTS and QUBT are known squeeze vehicles. The CIO has 2 years of trading
  history in these names but currently has no systematic signal for squeeze
  risk. Large unexplained moves in ENPH, UUUU, ASTS, FCEL in prior cycles
  were almost certainly squeeze-related but the system could not identify them.

Required data fields per ticker (where available):
  short_interest_pct_float    : short interest as % of float
  short_interest_shares       : total shares short
  days_to_cover               : short interest / avg daily volume
  short_interest_change_pct   : week-over-week change in short interest
  borrow_fee_pct              : annualised borrow cost (if available)
  short_interest_date         : date of last short interest report

Required computed field:
  squeeze_risk_score          : 0-100 composite score

Squeeze risk score formula (deterministic):

  Component 1 — Short Interest % Float (0-40 pts)
    >= 30% float short  : 40 pts
    20-30%              : 30 pts
    10-20%              : 20 pts
    5-10%               : 10 pts
    < 5%                : 0 pts

  Component 2 — Days to Cover (0-30 pts)
    >= 10 days          : 30 pts
    5-10 days           : 20 pts
    3-5 days            : 10 pts
    < 3 days            : 0 pts

  Component 3 — Short Interest Change (0-20 pts)
    Increasing >= 20%   : 20 pts (shorts piling in = fuel)
    Increasing 10-20%   : 10 pts
    Stable              : 5 pts
    Decreasing          : 0 pts

  Component 4 — Borrow Fee (0-10 pts)
    >= 100% annualised  : 10 pts (very hard to borrow)
    50-100%             : 7 pts
    20-50%              : 4 pts
    < 20%               : 0 pts

  squeeze_risk_score = sum of components (max 100)

Squeeze risk classification:
  80-100 : EXTREME — active squeeze conditions
  60-79  : HIGH    — squeeze risk elevated
  40-59  : MEDIUM  — monitor closely
  20-39  : LOW     — background risk only
  0-19   : MINIMAL — no squeeze signal

Data source:
  FINRA short interest data (bi-monthly, free, public)
  URL: https://www.finra.org/investors/learn-to-invest/
       advanced-investing/short-selling/short-interest
  Alternative: Moomoo short interest if API confirmed available
  Frequency: Update on FINRA release dates (1st and 15th of month)
             Store last known value between releases

Required additions to dataset_raw.json:
  short_interest[ticker] = {all fields above}

Required additions to research_report.txt:

  New section after L7 Capital Flow:

  L7b SHORT INTEREST & SQUEEZE RISK

  QBTS    XX.X%        X.X        XX/100        HIGH      INCREASING
  QUBT    XX.X%        X.X        XX/100        MEDIUM    STABLE
  [all satellite tickers]

  CIO Squeeze Alert (if score >= 60):
  ⚡ SQUEEZE ALERT: [ticker] squeeze_risk=HIGH (score XX/100)
