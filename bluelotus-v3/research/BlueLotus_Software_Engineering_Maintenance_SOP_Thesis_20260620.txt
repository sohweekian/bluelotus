ABSTRACT
-------------------------------------------------------------------------------
As BlueLotus grows from a personal investment tool into a self-learning
institutional cognitive digital organization, the software system becomes too
large to maintain through instinct, memory, or ad hoc patching.

The larger a software system becomes, the more dangerous casual bug fixing
becomes.

A bug is no longer only a broken line of code. In a large system, a bug may be:

  - a contradiction between modules,
  - stale data presented as live,
  - an agent hallucinating authority,
  - a dashboard showing outdated truth,
  - a database receiving redundant records,
  - a report contradicting governance,
  - a retry loop hiding runtime failure,
  - or a silent mismatch between what the CIO believes and what the machine
    remembers.

Therefore, BlueLotus requires a formal Software Engineering Maintenance SOP.

This thesis proposes that maintenance must itself become a governed layer of the
institution. Every bug must be observed, hypothesized, tested, refined, and
documented. Every failure must become memory. Every fix must preserve law.

The objective is not merely to clear bugs. The objective is to make BlueLotus
more truthful after every bug.

===============================================================================
1. INTRODUCTION
===============================================================================

In small software projects, bugs are often fixed by direct intuition:

  "This looks wrong. Change this line. Run it again."

That method works only while the system is small and the consequences are local.

BlueLotus V3 is no longer small.

It now contains:

  - deterministic operators,

  - Chief Strategist synthesis,
  - ACMS-COP operational layer,
  - Prospective Event Intelligence,
  - Shannon-Thorp refinement,

  Observe.
  Hypothesize.
  Work order.
  Patch.
  Test.
  Observe again.
  Refine.
  Document.
  Archive.

This is not bureaucracy.

This is institutional survival.

2. THE CORE PROBLEM: SYSTEM SIZE CREATES MAINTENANCE RISK
===============================================================================

As software grows, bugs become harder to reason about because the failure surface
expands.

A problem seen on the dashboard may originate from:

  - data ingestion,
  - stale JSON,
  - MySQL query,
  - publisher cache,
  - browser cache,
  - GitHub Pages propagation,
  - incorrect renderer logic,
  - incorrect source hierarchy,
  - or a correct source being displayed under the wrong label.

A problem seen in an agent report may originate from:

  - prompt starvation,
  - malformed JSON,
  - Ollama socket instability,
  - Qwen context overload,
  - retry prompt failure,
  - schema validator over-strictness,
  - weak role memory,
  - or deterministic governance not being injected.

A problem seen in a report may originate from:

  - stale dataset,
  - dashboard/report timestamp mismatch,
  - wrong renderer,
  - wrong source precedence,
  - old V2 path contamination,
  - or a governance layer not being called.

Therefore, the visible symptom is rarely the full cause.

The first rule of maintenance is:

  Do not patch the symptom until the system has been observed.

3. BLUELOTUS MAINTENANCE DOCTRINE
===============================================================================

The BlueLotus Software Engineering Maintenance SOP is built on ten laws.

LAW 1 — OBSERVE BEFORE PATCHING
-------------------------------------------------------------------------------
No engineer should patch before reading the relevant evidence.

Evidence includes:

  - failing test output,
  - latest cycle folder,
  - learning_loop_snapshot.json,
  - agent_errors,
  - LLM client logs,
  - publisher logs,
  - dashboard JSON,
  - report output,
  - database row state,
  - and current configuration.

The engineer must first answer:

  What failed?
  Where was it observed?
  When did it start?
  Which layer owns the truth?
  Which layer only displays the truth?

LAW 2 — FORM A HYPOTHESIS
-------------------------------------------------------------------------------
Every fix begins as a hypothesis, not as certainty.

The engineer should state:

  Observed symptom:
  Evidence:
  Hypothesis:
  Risk if wrong:
  Smallest safe test:

This prevents emotional or rushed patching.

LAW 3 — WRITE A WORK ORDER FOR NON-TRIVIAL BUGS
-------------------------------------------------------------------------------
When a bug touches governance, agents, reports, dashboard, database, execution
safety, or public publishing, it deserves a work order.

The work order should include:

  - problem,
  - affected files,
  - proposed solution,
  - safety invariants,
  - acceptance tests,
  - rollback notes,
  - and final report requirement.

LAW 4 — CHANGE ONE LOGICAL LAYER AT A TIME
-------------------------------------------------------------------------------
Do not mix unrelated changes.

Avoid combining:

  - prompt changes,
  - database changes,
  - dashboard rendering,
  - report naming,
  - governance rules,
  - and runtime process handling

unless the work order explicitly requires integration.

This makes causality observable.

LAW 5 — TEST NARROWLY FIRST
-------------------------------------------------------------------------------
Run the smallest relevant test first.

Examples:

  - one unit test for a classifier,
  - one renderer test,
  - one governance test,
  - one schema validation test,
  - one prompt validator,
  - one publisher dry run.

If the small test fails, do not run the whole system yet.

LAW 6 — TEST BROADLY AFTER LOCAL SUCCESS
-------------------------------------------------------------------------------
Once the local fix passes, run broader regression tests.

For BlueLotus, broad tests may include:

  - affected test file,
  - nearby module tests,
  - full pytest suite,
  - V3 grand cycle,
  - publisher run,
  - public endpoint verification.

LAW 7 — OBSERVE LOGS AFTER TESTING
-------------------------------------------------------------------------------
A passing command is not enough.

After the test, read the logs again.

Check:

  - Did new errors appear?
  - Did the same error disappear?
  - Did a new layer begin failing?
  - Did a background process remain stuck?
  - Did public output update?
  - Did contradiction governance detect new conflicts?

LAW 8 — REFINE WITHOUT EGO
-------------------------------------------------------------------------------
If the first hypothesis is wrong, revise it.

The engineer must never defend a bad fix because it was their fix.

The system is the teacher.
The logs are the evidence.
The tests are the court.

LAW 9 — RECORD THE FINAL KNOWLEDGE
-------------------------------------------------------------------------------
Every meaningful fix must end with a technical report.

The report should include:

  - what failed,
  - what evidence was observed,
  - the hypothesis,
  - what was changed,
  - what tests were run,
  - what errors remained,
  - what was refined,
  - final verification,
  - and safety invariants.

LAW 10 — PRESERVE BLUELOTUS LAW
-------------------------------------------------------------------------------
No maintenance action may violate:

  CIO_ONLY_MANUAL
  order_routing_enabled = false
  llm_order_generation = false
  broker execution disabled
  append-only memory
  no silent V2 contamination
  no hidden hardcoding
  no unrecorded doctrine overwrite
  no silent contradictions

4. THE BLUELOTUS BUG-SOLVING WORKFLOW
===============================================================================

The official maintenance workflow is:

STEP 1 — OBSERVE
-------------------------------------------------------------------------------
Read the failure surface.

For pipeline bugs:

  C:\bluelotus3\data\v3_cycles\<latest_cycle>\
  learning_loop_snapshot.json
  agent_errors\
  cycle_quality_summary.json
  governance\

For LLM bugs:

  C:\bluelotus3\logs\llm\
  Ollama process state
  model registry
  prompt audit
  retry records

For dashboard bugs:

  C:\bluelotus3\data\frontend\
  public GitHub endpoint
  publisher logs
  browser cache possibility

For report bugs:

  C:\bluelotus3\research\
  generated TXT / DOCX / XLSX
  renderer code
  dataset_raw.json
  governance outputs

STEP 2 — HYPOTHESIZE
-------------------------------------------------------------------------------
Write the likely cause.

Example:

  Symptom:
    Dashboard shows stale council cycle.

  Evidence:
    Publisher loaded latest cycle but public endpoint still shows older cycle.

  Hypothesis:
    GitHub Pages propagation delay or wrong push path.

  Test:
    Re-query public endpoint with cache-busting timestamp after delay.

STEP 3 — WORK ORDER
-------------------------------------------------------------------------------
For non-trivial bugs, write a short work order.

The work order is the contract between observation and action.

STEP 4 — PATCH
-------------------------------------------------------------------------------
Apply the smallest coherent change.

Avoid casual rewrites.
Avoid unrelated cleanup.
Avoid hidden constants.

STEP 5 — NARROW TEST
-------------------------------------------------------------------------------
Run the smallest test that proves the fix.

STEP 6 — BROAD TEST
-------------------------------------------------------------------------------
Run broader tests once the narrow test passes.

STEP 7 — REAL CYCLE TEST
-------------------------------------------------------------------------------
If the bug touches V3 intelligence flow, run:

  python -m orchestration.run_v3_grand_cycle

Confirm:

  validated_agent_reports
  agent_errors
  quality_summary
  contradiction_register
  cio_decision_strip
  learning_loop_snapshot

STEP 8 — LOG REVIEW
-------------------------------------------------------------------------------
Read logs after execution.

The absence of visible failure is not proof.
The absence of new error logs is stronger proof.

STEP 9 — REFINE
-------------------------------------------------------------------------------
If logs reveal new evidence, refine the fix.

STEP 10 — PUBLISH
-------------------------------------------------------------------------------
Only publish when local artifacts are clean enough.

If GitHub Pages is involved, verify the public endpoint after propagation.

STEP 11 — TECHNICAL REPORT
-------------------------------------------------------------------------------
Write the final report.

The report becomes institutional memory.

5. WHY THIS SOP MATTERS
===============================================================================

BlueLotus is becoming a digital organization, not a script collection.

A digital organization requires:

  - memory,
  - law,
  - audit trail,
  - evidence,
  - role discipline,
  - safety boundaries,
  - contradiction detection,
  - and repeatable engineering procedure.

Without maintenance SOP, the system will decay as it grows.

Symptoms of decay include:

  - duplicate logic,
  - stale dashboard data,
  - conflicting reports,
  - agent role drift,
  - database bloat,
  - unexplained regressions,
  - manual fixes nobody remembers,
  - and fear of changing the system.

The SOP prevents this.

It turns every bug into an opportunity for institutional learning.

6. MAINTENANCE AS INSTITUTIONAL LEARNING
===============================================================================

In ordinary software, bugs are embarrassing.

In BlueLotus, bugs are research material.

Every bug teaches:

  - where the system was brittle,
  - where assumptions were wrong,
  - where tests were missing,
  - where logs were insufficient,
  - where governance was silent,
  - and where the architecture must mature.

This is why bug reports must not be deleted emotionally.

They are part of the memory of the institution.

The correct attitude is:

  "The system revealed a weakness. We will observe it, understand it,
   correct it, test it, and record the lesson."

Codex, Claude Code, ChatGPT, and future AI engineering agents must not behave as

They must behave as institutional engineers.

Their duty is:

  - protect CIO authority,
  - protect deterministic truth,
  - protect database memory,
  - protect report consistency,
  - protect dashboard honesty,
  - protect execution safety,
  - and protect future engineers from confusion.

An AI engineer must not only answer:

  "Can I fix this?"

It must also answer:

  "How will the institution know what I changed, why I changed it,
   how I tested it, and what risks remain?"

===============================================================================
8. THE ROLE OF CONTRADICTION GOVERNANCE
===============================================================================

As BlueLotus grows, contradictions will become more common.

Examples:

  - Report says REVIEW but dashboard looks calm.
  - Agent says HOLD but deterministic operator blocks add-risk.
  - CIO policy wants gold miners but gold thesis blocks risk increase.
  - Public dashboard shows last valid cycle while latest runtime cycle failed.
  - Report uses live-sounding language for stale data.

Contradiction governance is therefore part of maintenance.

It tells engineers:

  "This system is not merely failing technically. It is disagreeing with itself."

That distinction is critical.

Technical errors break execution.
Contradictions break trust.

9. THE BLUELOTUS MAINTENANCE MATURITY MODEL
===============================================================================

LEVEL 0 — AD HOC PATCHING
-------------------------------------------------------------------------------
Bug appears.
Engineer changes code.
No test.
No report.
No memory.

This level is prohibited.

LEVEL 1 — LOCAL FIXING
-------------------------------------------------------------------------------
Bug appears.
Engineer fixes local symptom.
Some testing.
Little documentation.

Acceptable only for trivial cosmetic bugs.

LEVEL 2 — TESTED MAINTENANCE
-------------------------------------------------------------------------------
Bug appears.
Engineer writes/uses tests.
Fix is verified.

Minimum acceptable standard for BlueLotus.

LEVEL 3 — GOVERNED MAINTENANCE
-------------------------------------------------------------------------------
Bug appears.
Logs are read.
Hypothesis is written.
Work order is created.
Tests are run.
Report is written.

Required for V3 architecture work.

LEVEL 4 — SELF-LEARNING MAINTENANCE
-------------------------------------------------------------------------------
Bug appears.
System logs expose root cause.
Contradiction governance identifies institutional conflict.
Tests are added.
Fix is archived.
Future engineers inherit the lesson.

This is the BlueLotus target state.

===============================================================================
10. FINAL DOCTRINE
===============================================================================

The bigger the system becomes, the less acceptable instinctive patching becomes.

BlueLotus must maintain itself like an institution:

  - with evidence,
  - with memory,
  - with discipline,
  - with audit,
  - with humility,
  - and with law.

A bug is not merely a defect.

A bug is a message from the system saying:

  "There is something about me that is not yet understood."

The duty of the engineer is not to silence that message quickly.

The duty of the engineer is to understand it, repair it, test it, and preserve
the lesson for the future.

That is how BlueLotus becomes robust.

That is how a digital organization learns.

That is how software maintenance becomes institutional intelligence.
